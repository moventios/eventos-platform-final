## 1. Brand Voice

We are the **Reliable Coordinator** — calm, competent, direct, and outcome-focused.  
We speak like a senior ops professional who has run hundreds of events and knows exactly what matters.

### Voice Dimensions

| Dimension       | What It Means                                            | In Practice                                                                                            |
| --------------- | -------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| **Confident**   | We know what we're doing. No hedging.                    | "Run events without dropped balls." NOT "We try to help you manage..."                                 |
| **Clear**       | Simple language. Professionals want clarity, not jargon. | "See every booking conflict before it happens." NOT "Leverage our AI-powered spatial conflict engine." |
| **Human**       | We talk like people, not corporations.                   | "Your team spent 3 hours reconciling that. You shouldn't have to."                                     |
| **Trustworthy** | We prove claims with specifics, not adjectives.          | "Every transaction has a timestamp, actor ID, and trace." NOT "Best-in-class audit capabilities."      |
| **Focused**     | We're built for operations teams. Not everyone.          | "Built for teams that run real operations."                                                            |

## 2. Tone by Context

Tone shifts by situation — voice stays the same.

| Context              | Tone                          | Guidance                                               | Example                                                                                     |
| -------------------- | ----------------------------- | ------------------------------------------------------ | ------------------------------------------------------------------------------------------- |
| **Homepage Hero**    | Bold, confident               | Lead with the problem or outcome. Short and sharp.     | "Run events and projects without the chaos."                                                |
| **Feature Pages**    | Explanatory, precise          | Describe _what it does_ + _why it matters_ for ops.    | "Bookings are locked against conflicts at the database level — no double-booking possible." |
| **Error Messages**   | Calm, specific, helpful       | Name the problem. Explain why. Tell them what to do.   | "Room A102 is already booked 10:00–12:00 on July 3. Choose a different time or room."       |
| **Success States**   | Warm, brief, forward-looking  | Confirm what happened. Point to next step if relevant. | "Event created. Your team can see it in the shared calendar."                               |
| **Empty States**     | Encouraging, actionable       | Show context + primary action. Never just blank.       | "No events yet. Create your first one to get started."                                      |
| **Onboarding**       | Friendly, step-by-step        | Guide without overwhelming. Short steps.               | "Let's set up your first event. It takes about 3 minutes."                                  |
| **Documentation**    | Technical, example-first      | Lead with code/example. Explain after.                 | Show the code block first, then the explanation.                                            |
| **Trust / Security** | Formal, specific, data-backed | Precise claims. No vague adjectives.                   | "SOC 2 Type II certified. Annual penetration tests by [firm]."                              |
| **Pricing**          | Transparent, direct           | No tricks, no hidden anchors. State value clearly.     | "Free for your first 2 events. Pro from $X/month."                                          |
| **Support / Help**   | Patient, practical            | Assume the user is stuck. Get to the answer fast.      | Start with the solution, then explain the context.                                          |

## 3. Headline Formula Bank

### Problem-Frame Headlines (Top of Funnel)

Use when the audience is **problem-aware** but not yet solution-aware.

- "Your events run fine — until they don't."
- "47 spreadsheets. 3 tools. Still missed the venue conflict."
- "The coordination chaos that grows with every event you add."
- "Your team is good. Your tools are holding them back."
- "Reconciling event finances manually is not a compliance strategy."

### Solution / Category Headlines

Use when introducing the platform as a category or solution.

- "One system for every event, project, and the finance behind it."
- "From planning to audit — without switching tools."
- "The operations platform that actually shows you what's happening."
- "Coordinate. Execute. Prove."
- "Everything your operations team needs. Nothing they don't."

### Benefit Headlines (Conversion)

Use on feature pages and when driving trial/demo CTAs.

- "No more reconciling. No more chasing. No more surprises."
- "Run events the same way twice — then a hundred times."
- "When the CFO asks, you'll have the answer."
- "See status, conflicts, and financials in one view."
- "AI that assists. Humans that approve. Records that last."

### Feature-Specific Headlines

| Feature              | Headline                                                   |
| -------------------- | ---------------------------------------------------------- |
| Scheduling / Booking | "Zero booking conflicts — enforced at the database level." |
| Financial Ledger     | "Every transaction. Full history. No exceptions."          |
| Approval Workflows   | "Every approval captured. Every decision traceable."       |
| AI Copilot           | "AI that suggests. You decide. Always."                    |
| Access Passes        | "Issue, scan, and track passes — from one screen."         |
| Reports & Audit      | "Export everything. Prove anything."                       |
| Multi-tenant         | "Your data. Your rules. Your control."                     |

## 4. CTA (Call-to-Action) Copy Bank

### Primary CTAs

| Context               | CTA Text                           | Notes                                |
| --------------------- | ---------------------------------- | ------------------------------------ |
| Homepage hero         | **Start free**                     | No friction — trial, not credit card |
| Homepage secondary    | **Watch a demo**                   | 5-min video, no signup required      |
| Enterprise evaluation | **Request a demo**                 | Human-led, for complex evaluations   |
| Feature page          | **See it in action**               | Links to interactive demo or video   |
| Documentation entry   | **Read the docs**                  | Not "Learn more" — be specific       |
| Template download     | **Get the template**               | Be direct about what they receive    |
| Pricing page          | **Start free** / **Talk to sales** | Clear split by tier                  |
| Trust Center          | **View security details**          | Not "Learn more"                     |
| Customer story        | **Read [Company]'s story**         | Use real company name if possible    |
| Learning center       | **Start learning**                 | Active verb                          |
| Blog post bottom      | **See how the platform works**     | Bridge content → product             |

### Secondary CTAs (lower commitment)

- "Explore the docs →"
- "See all features →"
- "Download the playbook →"
- "Browse templates →"
- "Compare plans →"

## 5. Subheadline / Supporting Copy Patterns

### Format: Problem → Platform → Proof

```
[H1: Outcome or problem statement]
[Subhead: What the platform does to solve it, in ≤25 words]
[Micro-proof: One specific proof point or metric]
```

**Example:**

> **Run Events and Projects Without the Chaos**  
> One coordinated platform for scheduling, resources, finance, and audit — built for operations teams.  
> _500+ events coordinated. 99.9% uptime. No double-bookings._

### Format: Feature → Benefit → Proof

```
[Feature name or verb phrase]
[1–2 sentences: what it does + why it matters]
[Proof: specific metric, quote, or example]
```

**Example:**

> **Conflict-Free Booking**  
> Every venue, room, and resource booking is checked against the calendar at the moment of creation. Double-bookings are structurally impossible.  
> _"We haven't had a venue conflict since we moved to the platform." — Ops Director, [Company]_

## 6. Social Proof Copy Standards

### Testimonial Format

- Length: 25–50 words
- Must be **outcome-specific**, not generic ("We love it")
- Must include: name, role, company, industry tag

**Template:**

> "[Specific outcome in their words]. [Optional: before/after or metric.]"  
> — [First Name], [Role], [Company] · [Industry]

**Good example:**

> "Reconciliation used to take our finance team 3 days after every event. Now it's done by the time the event ends."  
> — Reza, CFO, Artha Events · Corporate Events

**Bad example (reject this):**

> "The platform is great. Very easy to use and the team is helpful."  
> — John, Manager, Company XYZ

### Customer Story Format

Structure: **Problem → Solution → Results**

1. Context: Who is the customer, what do they do, what was the challenge?
2. Problem: Specific pain (coordination chaos, audit failures, reconciliation manual)
3. Decision: Why they chose this platform
4. Solution: What they use (2–3 specific features)
5. Results: Measurable outcomes (time saved, errors reduced, audit passed)

## 7. Forbidden Patterns

### Words Never to Use in External Copy

- "Robust" — vague, overused
- "Seamless" — means nothing
- "Powerful" — every product claims this
- "Game-changing" — cringe
- "Revolutionary" — cringe
- "Innovative" — cringe
- "World-class" — unverified
- "Best-in-class" — unverified
- "Cutting-edge" — dated
- "Leverage" — corporate jargon
- "Synergy" — corporate jargon
- "Solution" as a noun (e.g., "our solution helps") — vague; name the thing

### Architecture Terms Never to Use in External Copy

These are for internal engineering docs ONLY. Never expose in UI, marketing, or customer-facing content:

| Internal term            | External translation                                 |
| ------------------------ | ---------------------------------------------------- |
| Immutable ledger         | "Full audit trail" / "Records that can't be altered" |
| Hexagonal architecture   | (never mention)                                      |
| DDD / Bounded Context    | (never mention)                                      |
| L-06 guardrail           | "AI that requires human approval"                    |
| Domain Events            | (never mention)                                      |
| RLS (Row Level Security) | "Your data is isolated from other tenants"           |
| Idempotency key          | (never mention — it's an implementation detail)      |
| Layer-1 Constitution     | (never mention)                                      |
| Tenant sovereignty       | "Your data, your control" / "No vendor lock-in"      |

## 8. Numbers & Data Standards

- Always use **specific numbers** over vague language
  - ✅ "3-day reconciliation → same-day"
  - ❌ "Dramatically faster reconciliation"
- Amounts: Always use `<AmountDisplay>` component in UI. Never raw floats.
- Metrics in copy: round to meaningful precision
  - ✅ "99.9% uptime"
  - ✅ "500+ events coordinated"
  - ❌ "99.97293% uptime" — too precise, feels unnatural
- Avoid superlatives without proof: "largest," "fastest," "best" → only if verifiable

## 9. Localization Notes (Indonesia Market)

- **Language**: Primary content in Bahasa Indonesia for Indonesia market pages
- **Tone in Bahasa**: Formal but approachable. Use "Anda" not "kamu". Mirror the professional tone.
- **Currency**: IDR with `Rp` prefix, Indonesian number formatting (`Rp 1.500.000`, not `Rp1,500,000`)
- **Payment references**: Mention Midtrans, Xendit, QRIS alongside global options (Stripe)
- **Compliance**: Reference Indonesian regulations (PBI OJK) alongside GDPR where relevant
- **Examples**: Use Indonesian company/event contexts in case studies
